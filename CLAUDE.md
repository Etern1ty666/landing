## 1. О проекте

Лендинг-сайт. Одностраничник с потенциалом роста (роутер уже подключаем под будущее расширение).

**Стек:**
- React 18
- TypeScript (strict)
- Vite
- React Router (планируется)

**Команды:**
```bash
npm install        # установка зависимостей
npm run dev        # дев-сервер на http://localhost:5173
npm run build      # прод-сборка в dist/
npm run preview    # превью прод-сборки
```

## 2. Архитектура — Feature-Sliced Design (FSD)

Проект построен по **FSD**. Это не пожелание — это жёсткое правило. Любое нарушение FSD должно быть либо явно согласовано с пользователем, либо исправлено.

Документация методологии: https://feature-sliced.design/

### 2.1. Слои (layers)

Сверху вниз — от высокоуровневого к низкоуровневому:

```
src/
├── app/          # инициализация: провайдеры, роутер, глобальные стили
├── pages/        # страницы-роуты (композиция widgets/features/entities)
├── widgets/      # самостоятельные UI-блоки (Header, Footer, HeroSection)
├── features/     # пользовательские действия (SubscribeForm, ContactForm)
├── entities/     # бизнес-сущности (Article, Plan, User)
└── shared/       # переиспользуемое без привязки к бизнесу (UI-kit, utils, api)
```

### 2.2. Правило импортов (КРИТИЧНО)

**Импорт идёт ТОЛЬКО сверху вниз. Никогда вбок, никогда вверх.**

| Слой       | Может импортировать из                                     |
|------------|------------------------------------------------------------|
| `app`      | `pages`, `widgets`, `features`, `entities`, `shared`       |
| `pages`    | `widgets`, `features`, `entities`, `shared`                |
| `widgets`  | `features`, `entities`, `shared`                           |
| `features` | `entities`, `shared`                                       |
| `entities` | `shared`                                                   |
| `shared`   | — ничего из проекта (только внешние библиотеки)            |

**Запрещено:**
- Импорт между слайсами одного слоя (одна `feature` не знает о другой `feature`).
- Импорт «вверх» (`shared` не знает про `entities`, `entities` не знает про `features` и т.д.).
- Прямой импорт во внутренности слайса (см. Public API ниже).

Если две фичи должны взаимодействовать — общение идёт через `entities` или результат поднимается в `widget`/`page`.

### 2.3. Слайсы (slices)

Внутри `pages/widgets/features/entities` код делится на слайсы — папки по бизнес-доменам:

```
widgets/
├── header/
├── footer/
└── hero-section/
```

Слайс именуется в `kebab-case`. Один слайс = одна осмысленная единица.

### 2.4. Сегменты (segments)

Внутри слайса — стандартные сегменты (создаём только нужные):

```
widgets/header/
├── ui/          # React-компоненты слайса
├── model/       # состояние, селекторы, бизнес-логика, типы домена
├── api/         # сетевые запросы
├── lib/         # утилиты, узко применимые в слайсе
├── config/      # константы слайса
└── index.ts     # Public API
```

Слой `shared` устроен похоже, но без слайсов — сразу сегментами:
```
shared/
├── ui/          # UI-kit (Button, Input, Container)
├── api/         # базовый http-клиент
├── config/      # env, конфиг
├── lib/         # утилиты
├── hooks/       # общие хуки
├── types/       # общие TS-типы
├── constants/   # роуты, ключи
└── assets/      # картинки, иконки, шрифты
```

### 2.5. Public API

Каждый слайс **обязан** иметь `index.ts` — это его публичный интерфейс.

**Снаружи слайс импортируется ТОЛЬКО через `index.ts`:**

```ts
// ✅ правильно
import { Header } from '@/widgets/header';

// ❌ запрещено
import { Header } from '@/widgets/header/ui/Header';
```

В `index.ts` ре-экспортируем только то, что должно быть видно снаружи. Внутренние компоненты, хелперы, типы — остаются приватными.

### 2.6. Алиас путей

В проекте используется алиас `@/*` → `src/*`. Всегда импортируй через `@/`, не относительными путями между слоями.

```ts
// ✅
import { Button } from '@/shared/ui';

// ❌
import { Button } from '../../../shared/ui';
```

## 3. Конвенции кода

### Именование
- **Слайсы и папки**: `kebab-case` (`hero-section`, `subscribe-form`).
- **Компоненты и файлы компонентов**: `PascalCase` (`Header.tsx`).
- **Хуки**: `useSomething.ts`.
- **Утилиты, типы, константы**: `camelCase.ts`.
- **TS-типы и интерфейсы**: `PascalCase`, без префикса `I`.

### Компоненты
- Функциональные компоненты + хуки. Классы не используем.
- Один компонент = один файл. Имя файла = имя компонента.
- Props типизируются явно: `type HeaderProps = { ... }`.
- Не экспортировать всё подряд из `index.ts` слайса — только то, что реально используется снаружи.

### TypeScript
- Режим `strict` включён. Никаких `any` без явного обоснования в комментарии.
- Для пропсов и публичных API — явные типы. Внутри функций — можно положиться на вывод типов.

### Стили
- Договариваемся отдельно (CSS-modules / Tailwind / styled). Пока стилей нет.

## 4. Где что создавать (cheat sheet для ассистента)

Перед тем как создать новый файл — сверься с таблицей:

| Что создаём                                       | Куда                                              |
|---------------------------------------------------|---------------------------------------------------|
| Новая страница-роут                               | `src/pages/<name>/`                               |
| Большой самостоятельный блок UI (Header, секция)  | `src/widgets/<name>/`                             |
| Форма/действие пользователя (подписка, фильтр)    | `src/features/<name>/`                            |
| Бизнес-сущность (карточка статьи, тарифа)         | `src/entities/<name>/`                            |
| Кнопка, инпут, контейнер — UI-примитив            | `src/shared/ui/<Component>/`                      |
| Общий хук (`useMediaQuery`, `useDebounce`)        | `src/shared/hooks/`                               |
| Общая утилита (formatDate, cn)                    | `src/shared/lib/`                                 |
| HTTP-клиент, базовый fetch                        | `src/shared/api/`                                 |
| Конфиг роутера                                    | `src/app/router/`                                 |
| Глобальный провайдер (Theme, Router, QueryClient) | `src/app/providers/`                              |
| Layout с `<Outlet />`                             | `src/app/providers/` или отдельный widget         |
| Картинка/иконка/шрифт                             | `src/shared/assets/{images,icons,fonts}/`         |

Если непонятно, куда положить — **спроси у пользователя**, не угадывай.

## 5. Чек-лист перед сдачей изменений

Перед тем как сказать «готово», убедись:

1. [ ] Нет нарушений правила импортов (сверху вниз, не вбок).
2. [ ] Нет импортов во внутренности слайса (только через `index.ts`).
3. [ ] Все пути идут через алиас `@/`.
4. [ ] Новый слайс имеет `index.ts` с public API.
5. [ ] Именование папок — `kebab-case`, компонентов — `PascalCase`.
6. [ ] `npm run build` проходит без ошибок TypeScript.
7. [ ] Нет `any` без обоснования.
8. [ ] Нет мёртвого кода, закомментированных блоков, лишних комментариев.

## 6. Поведенческие правила для AI

- **Не создавай файлы, пока пользователь не подтвердил подход.** Сначала — план, потом — код.
- **Не переусложняй.** Лендинг — это лендинг. Не тащи сюда Redux, если хватает локального state.
- **Не нарушай FSD ради скорости.** Лучше спросить, чем класть компонент «куда-нибудь».
- **При сомнениях — задай вопрос**, а не угадывай.
- **Язык общения с пользователем — русский.** В коде — английский (имена, комментарии).
- **Не добавляй комментарии в код без необходимости.** Хорошие имена > комментарии.

## 7. Что НЕ делать

- ❌ Создавать `utils/`, `helpers/`, `common/` в корне `src/` — это анти-паттерн FSD.
- ❌ Класть бизнес-компоненты в `shared/ui` (там только примитивы без знания о домене).
- ❌ Импортировать одну фичу из другой.
- ❌ Делать «универсальный» компонент с десятком пропсов вместо двух специализированных.
- ❌ Молча менять архитектурное решение из этого файла. Если считаешь, что правило неудачное — обсуди с пользователем и обнови `CLAUDE.md`.
